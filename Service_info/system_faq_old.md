Интеграция с iiko (OLAP)
Для синхронизации доставок и выручки курьеров используются OLAP-отчеты iiko Server (Resto API).

Основные поля для отчета DELIVERIES:
Тип отчета: DELIVERIES

Обязательный фильтр: OpenDate.Typed (Учетный день)

Группировка (groupByRowFields):

Delivery.Number — Номер доставки (используется как ID)

Delivery.Courier — ФИО курьера

Delivery.Street — Улица

Delivery.Region — Зона доставки (регион)

Delivery.ActualTime — Фактическое время доставки

Delivery.ExpectedTime — Ожидаемое время доставки

Delivery.SendTime — Время отправки (используется для CookingFinishTime)

Агрегация (aggregateFields):

fullSum — Полная сумма заказа

Особенности:
Старые поля вида Courier.Name или OrderSum вызывают ошибку 400 Bad Request. Всегда используйте префикс Delivery. для отчета по доставкам.

Для финансовых отчетов (выручка) используется тип SALES, где поля могут отличаться (например, DishDiscountSum.AfterDiscount).

Права доступа API iiko Cloud
Для полноценной работы интеграции API-логин в iiko Cloud должен обладать определенными правами.

Проблема с /api/1/employees
Эндпоинт /api/1/employees требует прав на управление персоналом (Staff Management). Если у API-логина нет этих прав, iiko возвращает ошибку 401 Unauthorized с описанием Right api/1/employees is not allowed for this ApiLogin.

Механизм Fallback
В IikoService.get_employees реализован механизм автоматического переключения:

Система пытается получить полный список сотрудников через /api/1/employees.

Если возвращается ошибка прав (401/403), система переключается на эндпоинт /api/1/employees/couriers, который обычно доступен для большинства API-логинов.

В логи выводится WARNING о переключении, но критическая ошибка ERROR подавляется, так как это штатная ситуация.

Рекомендации по настройке iiko
Если вам нужен полный список сотрудников (не только курьеров), убедитесь, что в iikoOffice/iikoChain для используемого пользователя API назначена роль с правами на просмотр списка сотрудников.

Сотрудники (Resto API vs Cloud API):

Cloud API: Эндпоинт /api/1/employees часто ограничен (401/403). Система использует fallback на /api/1/employees/couriers.

Server API (Resto): Используется методом sync_employees_full для полной синхронизации штата (администраторы, повара и т.д.). Требует настройки resto_login и resto_password в админ-панели.

Работа с временем iiko Cloud API: iiko Cloud API v2 присылает время в формате ISO.

Решение: Все данные из Iiko Cloud API (Transport) теперь принимаются как UTC 0 (согласно суффиксу 'Z'). В базе данных время хранится в UTC. В админ-панели время автоматически корректируется согласно настройкам часового пояса в IikoSettings. Это исправляет проблему 5-часового смещения. (Обновлено 22.04.2026)

Синхронизация полных адресов (24.04.2026)
В систему внесено важное улучшение для решения проблемы "неполных адресов" из iiko:

Объединение полей: Теперь IikoSyncService ищет компоненты адреса (дом, квартира, подъезд, этаж, домофон) одновременно в объектах address и deliveryPoint ответа iiko. Это исключает потерю данных, если iiko заполнила квартиру в одном месте, а улицу в другом.

Умный формат Line1: Даже если в настройках выбран формат "Одной строкой", система теперь проверяет, содержит ли эта строка упоминание квартиры, подъезда и т.д. Если данные есть в отдельных полях, но отсутствуют в строке line1, они автоматически дописываются в конец через запятую.

Сохранение в БД: Все компоненты адреса теперь гарантированно сохраняются в соответствующие колонки таблицы orders (flat, entrance, floor, doorphone), что позволяет фронтенду корректно отображать их в деталях заказа.

Автоматическое создание карточек гостей (26.04.2026)
Реализован механизм автоматического создания и обновления карточек гостей при поступлении заказов из любых источников (iiko Cloud Webhooks, Сайт, Telegram-бот).

Логика: При поступлении заказа проверяется наличие гостя по номеру телефона. Если гостя нет, создается новая карточка. Если гость есть, обновляются его данные (имя, фамилия, telegram_id) и дата последнего заказа.

Заметки (notes): В поле заметки автоматически добавляется информация о номере заказа, дате и сумме для истории.

Фоновая синхронизация: Сразу после создания или обновления карточки в БД, система ставит задачу в Celery (sync_single_customer_task) для получения актуальных данных из iiko Cloud (бонусы, категории, адреса).

Файлы: backend/app/services/iiko_sync_service.py (метод ensure_customer_from_order), backend/app/api/orders.py.
