## Инфраструктура фоновых задач (Celery)

Для выполнения длительных операций (массовая синхронизация, импорт из файлов, рассылки) используется **Celery** с **Redis** в качестве брокера сообщений.

### Компоненты:

- **Broker**: `redis://redis:6379/0` (контейнер `foodtech-redis`)
- **Worker**: Отдельный Docker-контейнер `foodtech-worker`, запускающий `celery -A app.core.celery_app worker`.
- **App**: Конфигурация находится в `backend/app/core/celery_app.py`.
- **Задачи**: Определены в `backend/app/tasks/customer_tasks.py`.

## Интеграция с iiko (OLAP)

Для синхронизации доставок и выручки курьеров используются OLAP-отчеты iiko Server (Resto API).

### Основные поля для отчета DELIVERIES:

- **Тип отчета**: `DELIVERIES`
- **Обязательный фильтр**: `OpenDate.Typed` (Учетный день)
- **Группировка (groupByRowFields)**:
  - `Delivery.Number` — Номер доставки
  - `Delivery.Courier` — ФИО курьера
  - `Delivery.Street` — Улица
  - `Delivery.Region` / `Delivery.Zone` — Зона доставки
  - **[EXTENDED]** `Delivery.House`, `Delivery.Flat`, `Delivery.Entrance`, `Delivery.Floor`, `Delivery.Doorphone` — Компоненты адреса (с авто-откатом при отсутствии поддержки).
- **Агрегация (aggregateFields)**:
  - `fullSum` — Полная сумма заказа
  - `UniqOrderId` — Уникальный ID заказа (обязательно для предотвращения 422 ошибки)

### Логика отказоустойчивости:
1. **Умный запрос**: Сначала пробуем v2 с расширенными полями адреса.
2. **Retry (400)**: Если поля не поддерживаются, повторяем без них.
3. **Fallback (404)**: Если v2 API недоступен, переключаемся на v1 (GET).
4. **Parsing**: Если компоненты адреса не получены от API, используется regex-парсинг строки `Delivery.Address`.

## Sync API (Employees & Couriers)

Для управления данными из iiko вручную реализованы следующие эндпоинты (FastAPI):

- `POST /api/v1/employees/sync`: Запуск полной фоновой синхронизации (Сотрудники + Смены + Доставки). Используется кнопка на главной странице сотрудников.
- `POST /api/v1/employees/sync/couriers`: [NEW] Запуск фоновой синхронизации **только доставок курьеров**. Используется кнопка во вкладке "Отчеты".
- `POST /api/v1/employees/sync/full`: [NEW] Синхронный вызов (ожидает завершения). Рекомендуется использовать только для отладки.

**Параметр `days`**: Все эндпоинты принимают `days` (int). Для курьеров по умолчанию — 7 дней, для полной синхронизации — 14.

## Синхронизация клиентов (Customers)

### 1. Массовая синхронизация (Рекурсивная схема)

- **Размер пачки**: 50 клиентов.
- **Пауза**: 50 секунд между пачками.
- **Логика**: Рекурсивный вызов Celery задач с инкрементом `skip`.
- **Управление**: Можно поставить на паузу или отменить через страницу "Фоновые процессы".

### 2. Импорт клиентов из файлов (XLSX, XML)

- **Эндпоинт**: `POST /api/v1/customers/import`
- **Логика**: Файл загружается в `backend/temp_imports/`, затем запускается фоновая задача.
- **Очистка**: Файлы можно удалять вручную в разделе "Фоновые процессы".

### 3. Авто-создание при заказе (Обновлено 26.04.2026)

В `iiko_sync_service.py` реализован метод `ensure_customer_from_order`:

- **Триггер**: Поступление заказа с сайта, из бота или через вебхук iiko.
- **Проверка**: Поиск по номеру телефона в таблице `customers`.
- **Действие**: Если гостя нет, создается новая карточка (Имя, Фамилия, Телефон). Если есть — обновляются данные (имя, telegram_id) и дата последнего заказа.
- **История**: В поле `notes` автоматически дописывается информация о номере заказа, дате и сумме.
- **Синхронизация**: Сразу запускается фоновая задача `sync_single_customer_task` для получения из iiko Cloud актуальных бонусов, категорий и адресов.

### 5. Расширенная карточка клиента (Laravel + Vue.js) (Обновлено: 26.04.2026)

- **Вкладка "Бонусы"**: Отображает текущий баланс кошельков и полную историю транзакций из таблицы `client_bonus_history`. Добавлена кнопка принудительной синхронизации истории транзакций напрямую из iiko.
- **Вкладка "Адреса"**: Список всех адресов, на которые клиент заказывал доставку (`client_addresses_history`), с указанием даты последнего использования и количества заказов.
- **Группа Риска**: Автоматическое определение по ключевым словам в категориях iiko ("риск", "стоп", "черный", "блок"). Статус отображается ярким баннером в карточке. Также используется флаг `should_be_checked_for_risk` из Cloud API.
- **История Заказов (12 мес)**: [ОПТИМИЗИРОВАНО 27.04.2026] Загрузка истории доставок из iiko Cloud.
  - **3 кнопки синхронизации**: Выбор периода (1, 3, 6 месяцев). При синхронизации данные объединяются аддитивно (новые добавляются, старые не удаляются).
  - **Кэширование**: Данные хранятся в поле `orders_history` (jsonb) с флагами `exists_locally` и `status`.
  - **Интеграция**: При клике на заказ в истории, админка проверяет наличие этого заказа в локальной базе (`orders.iiko_order_id`). Если заказ найден, открывается полная локальная карточка заказа со всеми деталями. Если нет — открывается упрощенное окно с деталями напрямую из истории iiko.
  - **Поиск и фильтрация**: Добавлен живой поиск по номеру заказа, дате или UUID прямо внутри вкладки "Заказы".
- **Заметки iiko**: Поле `iiko_notes` синхронизируется с комментарием гостя в iiko и доступно для редактирования.


### 7. Обслуживание и объединение базы (Maintenance) [Оптимизировано 28.04.2026]

Для поддержания чистоты базы данных и обеспечения высокой производительности реализованы следующие улучшения:

- **Стабилизация IikoSyncService**:
  - Полностью устранены зависания событийного цикла (Event Loop lag). Тяжелые SQL-операции (слияние сотен клиентов, агрегация статистики) теперь выполняются в отдельных потоках через `asyncio.to_thread`.
  - Внедрена ленивая инициализация `asyncio.Lock` для управления токенами iiko, что предотвращает ошибки `RuntimeError` при запуске приложения.
  - Оптимизирована структура файла `iiko_sync_service.py`: удалены дубликаты методов, исправлены отступы и восстановлена целостность кода.
  
- **Безопасность сессий**:
  - Все фоновые задачи теперь используют `SessionLocal` для корректного управления жизненным циклом соединений с БД, исключая утечки и конфликты в асинхронной среде.

- **Поиск без ID**: Эндпоинт `/api/v1/customers/maintenance/check-ids` находит клиентов, у которых не привязаны `uid` или `iiko_id`.

- **Объединение по UID**:
  - Поиск групп с одинаковыми `uid` из iiko.
  - Слияние всех связанных данных (заказы, адреса, телефоны, бонусы) в один мастер-аккаунт.
  - Удаление дублирующих записей.
  - **Производительность**: Операции слияния выполняются атомарно с периодическими коммитами для снижения нагрузки на БД.

- **Объединение по Телефону**:
  - Поиск клиентов с одинаковыми номерами телефонов.
  - Объединение данных и принудительная синхронизация финального профиля из iiko Cloud.
  - Статистика заказов агрегируется.

- **Логирование**: Все операции объединения фиксируются в локальных логах `/Service_info/log/YYYY-MM-DD/HH-MM/stabilization_sync.log`.

## Известные ошибки и их решение (Troubleshooting)

### 1. Ошибка "Unexpected token '<', \"<html>...\"" (JSON Parse Error)

**Проблема**: Frontend ожидает JSON, но получает HTML (502/500).
**Решение**: Перезапуск бэкенда и Nginx.
`docker-compose restart backend worker && docker exec foodtech-nginx nginx -s reload`

### 2. Ошибка iiko Cloud API 400 (Bad Request)

**Проблема**: `Validation_CommonValidationError` при создании/обновлении клиента.
**Решение**: В `iiko_service.py` добавлена очистка payload от `None`. Поля `birthday` передаются строго в формате `yyyy-MM-dd HH:mm:ss.fff`.

### 3. Ошибка 500 при загрузке адресов гостя

**Проблема**: `Unhandled exception: last_used_at` в `customers.py`.
**Решение**: В модели `GuestAddress` нет поля `last_used_at`. Сортировка изменена на `created_at`.

### 4. Некорректное отображение категорий лояльности

**Проблема**: Категории отображались как JSON-массив или пустая строка.
**Решение**: Во фронтенде (`CustomerDetailModal.vue`, `clients/index.vue`) добавлен парсинг строки через запятую ("VIP, Постоянный").

## Справочник API iiko Cloud (Loyalty)

### 1. Расчет лояльности (`/api/1/loyalty/iiko/calculate`)

- **Метод**: POST
- **Назначение**: Расчет скидок, бонусов и подарков перед созданием заказа.
- **Ключевые поля запроса**:
  - `order`: Объект заказа (items, payments, customer).
  - `coupon`: Номер примененного купона.
  - `applicableManualConditions`: UUID ручных условий.

### 2. Информация о клиенте (`/api/1/loyalty/iiko/customer/info`)

- **Метод**: POST
- **Поиск**: По `phone` (type="phone").
- **Важные поля ответа**:
  - `walletBalances`: Массив кошельков с балансами (бонусы).
  - `categories`: Категории гостя в iiko.
  - `birthday`: Формат `yyyy-MM-dd HH:mm:ss.fff`.

### 3. Создание/Обновление клиента (`/api/1/loyalty/iiko/customer/create_or_update`)

- **Метод**: POST
- **Обязательные поля**: `organizationId`, `phone` или `id`.
- **Формат даты**: `yyyy-MM-dd HH:mm:ss.fff`.

### 4. Работа с категориями

- `customer_category`: Список всех категорий организации.
- `customer_category/add`: Добавить категорию гостю.
- `customer_category/remove`: Удалить категорию.

### 5. Транзакции и баланс

- `customer/transactions/by_date`: История операций по кошельку.
- `customer/wallet/topup`: Пополнение баланса (начисление).
- `customer/wallet/chargeoff`: Списание баланса.
- `customer/wallet/hold`: Удержание бонусов (при создании заказа).

### 6. Коммуникации

- `message/send_sms`: Отправка SMS через шлюз iiko.
- `message/send_email`: Отправка Email.

- **Ошибка 403 (License enhancement is required)**:
  - **Причина**: Отсутствие лицензии на модуль "iiko Server API" (Resto API).
  - **Решение**: Активация лицензии через iikoOffice или дилера. Без этого модуля доступ к `/resto/api/*` (включая OLAP v2) закрыт.
- **Ошибка 400 (Unknown OLAP field 'Delivery.House')**:
  - **Причина**: Поле `Delivery.House` не поддерживается в текущей версии iiko в отчете `DELIVERIES`.
  - **Решение**: Использовать `Delivery.Address` и парсить его в коде. Поля `Delivery.Number`, `Delivery.Courier`, `Delivery.Street` работают корректно.

### 8. Ошибки прав доступа (Rights) iiko Cloud (Обновлено: 27.04.2026)

- **Ошибка 401 (Right api/1/... is not allowed for this ApiLogin)**:
  - **Причина**: У API-пользователя (ApiLogin) в iiko Cloud нет прав на конкретный метод (чаще всего "Сотрудники" или "Курьеры").
  - **Административное решение**: Зайти в ЛК iiko Cloud -> Настройки -> API Login -> Редактировать -> Проставить галочки прав.
  - **Техническое решение (Внедрено)**: В `iiko_service.py` добавлен автоматический fallback на Resto API для получения списка сотрудников, если Cloud API возвращает ошибку прав.

### 9. Логика автоматической регистрации гостей (Обновлено: 27.04.2026)

- **Файл**: `backend/app/services/iiko_sync_service.py` -> `ensure_customer_from_order`.
- **Процесс**:
  1. При получении заказа (через вебхук или синхронизацию) проверяется телефон в базе `customers`.
  2. Если гостя нет — создается новая карточка. Сохраняются: Имя, Фамилия, Телефон, Дата заказа (как дата последнего заказа) и детальная информация о заказе в `iiko_notes`.
  3. Для существующих гостей обновляется дата последнего заказа и статистика (кол-во, сумма).
  4. **Фоновая синхронизация**: Сразу после сохранения в БД запускается задача Celery `sync_single_customer_task`, которая подтягивает из iiko Cloud Loyalty баланс бонусов, категории гостя и другие детали профиля.
- **Логирование**: В логах искать метки `==> Customer Sync Trigger` и `==> REGISTRATION: New guest`.

### 8. Авто-создание карточки гостя (Алгоритм)

- **Проверка**: При каждом заказе (вебхук или API) вызывается `ensure_customer_from_order`.
- **Поиск**: Сначала по основному телефону, затем по таблице `guest_phones`.
- **Данные**: Сохраняются `name`, `surname`, `phone`, `email`.
- **Заметки**: В `notes` и `iiko_notes` пишется расширенная информация: номер заказа, дата, сумма и состав блюд.
- **Дата заказа**: Поле `last_order_date` обновляется временем создания заказа из iiko.
- **Синхронизация**: После сохранения в БД всегда запускается фоновая задача для подтягивания данных из iiko Cloud (бонусы, категории).
- **Маппинг OLAP**: Используется `UniqOrderId` для ID последнего заказа.

### 12. Нормализация телефонов (Обновлено: 27.04.2026) [NEW]

Для обеспечения уникальности и корректной работы с iiko Cloud внедрена строгая нормализация:

- **Единый формат БД**: Все номера хранятся в формате `+79991234567` (с одним знаком `+`).
- **Формат iiko API**: При передаче в iiko Cloud плюс удаляется (`79991234567`).
- **Автоматизация**: При каждом создании/обновлении клиента (через API или синхронизацию заказов) вызывается `normalize_phone`, исправляющая форматы `8...`, `++7...`, `7...` и удаляющая лишние символы.
- **Миграция**: Реализован скрипт `backend/scripts/sanitize_customers.py` для очистки базы от дублей, возникших из-за разного написания номеров (напр. `+7...` и `++7...`).
- **Фронтенд**: Во всех списках и формах (Clients/index.vue) убран жестко прописанный префикс `+`, так как номер теперь всегда содержит его в данных.

## 14. Стабилизация инфраструктуры синхронизации (Non-blocking I/O) [NEW 28.04.2026]

Для предотвращения блокировок основного event loop и устранения "зависаний" админ-панели при массовых операциях проведена глубокая оптимизация `iiko_sync_service.py`:

- **Offloading в потоки**: Все тяжелые операции с базой данных (SQLModel/SQLAlchemy) внутри асинхронных методов вынесены в фоновые потоки через `asyncio.to_thread`.
- **Разделение логики**: Методы синхронизации меню, номенклатуры и слияния клиентов разделены на асинхронные обертки (для API вызовов) и синхронные хелперы (для работы с БД).
- **Слияние клиентов**: Процесс поиска и объединения дублей по UID и телефону теперь выполняется без блокировки сервера, что позволяет администраторам продолжать работу во время очистки базы.
- **Безопасность транзакций**: Все синхронные хелперы используют `session.commit()` и `session.flush()` внутри потока, обеспечивая целостность данных.
